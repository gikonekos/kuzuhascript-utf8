# 20年前のCGI掲示板を、令和のサーバーで動かす話

> 「動かないコードより、動かない設定を疑え」を地でいく、
> とある土曜の昼下がり（正確には1時間19分）の記録。

## これは何の話？

`bbs.cgi` という、Perlで書かれた昔ながらの掲示板CGIスクリプトを、
Oracle CloudのUbuntuサーバー上で動かそうとしたら、思いのほか
手こずった――というだけの話です。日本時間の土曜の昼過ぎ、
14時前後の出来事です（サーバーのログはUTCで記録されるため、
以下の時刻表記はすべて日本時間＝JSTに直しています）。

大事なポイントが一つ。

> **掲示板としての動作ロジックには、一切手を加えていません。**

古いスクリプトをそのまま今のサーバーで動かすのがゴールで、
「動かないなら書き直そう」ではなく「動くようにサーバー側を整える」
という方針（＝**動態保存**）で進めています。骨董品を修理するとき、
部品を現代の新品に交換するのではなく、動くように整備する感じに
近いかもしれません。

ただし正直に言うと、原因を切り分けるための実験として、
スクリプトの冒頭に数行だけコードを足しており、しかもこれは
**今も削除できていません**（詳しくは最後の「おまけ」参照）。
「何もしていない」は言い過ぎで、正しくは「本来の掲示板としての
ロジックには手を入れていない」です。

## 事件簿：CGIが動くまでの一部始終

### 第1話：ページを開いたら、料理ではなくレシピカードが出てきた（13:54）

ブラウザで掲示板のURL（`/cgi-bin/bbs.cgi`）を開くと、掲示板の画面
ではなく、**Perlのソースコードがそのまま**表示されました。

これはたとえるなら、レストランで「オムライスください」と頼んだら、
できあがったオムライスではなく**オムライスの作り方のレシピカード**
を渡されたようなものです。

CGIというのは本来、「このファイルは"実行"するプログラムですよ」と
サーバーに伝えることで、プログラムを動かした**結果**（HTML）だけを
お客さん（ブラウザ）に見せる仕組みです。ソースコードがそのまま
見えてしまうということは、サーバーが「これは実行するプログラム」だと
認識していなかった、ということになります。

### 第2話：サーバーが違う引き出しを探していた（14:14）

設定を見直して一度サーバーを再起動したところ、今度は404エラー
（ページが見つかりません）になりました。エラーログにはこう
残っていました。

```
AH01264: script not found or unable to stat: /usr/lib/cgi-bin/bbs.cgi
```

実は、Ubuntu上のApache（サーバーソフト）には、あらかじめ
「CGIプログラムは`/usr/lib/cgi-bin/`の引き出しに入れてね」という
デフォルトのお約束がありました。でも実際にbbs.cgiを置いたのは
`/var/www/html/cgi-bin/`という**別の引き出し**。

サーバーからすれば「言われた引き出しを開けたら空だった」だけの
ことで、これは至って素直な反応です。「探す場所はこっちだよ」と
`ScriptAlias`という設定でサーバーに教え直してあげる必要があります
（これはApache側の**設定**の話で、bbs.cgi自体の中身とは関係
ありません）。

### 第3話：見えない文字と、古い地図のせいで動けない（14:22〜14:29）

引き出しの場所は正しく教えたのに、今度は500エラー（サーバー内部
エラー）に変わりました。

```
AH01241: exec of '/var/www/html/cgi-bin/bbs.cgi' failed
(No such file or directory)
End of script output before headers: bbs.cgi
```

これはスクリプトの**1行目**が原因でした。Perlスクリプトの先頭には
「このプログラムはこの場所にあるPerlで動かしてね」という一行
（シバンと呼ばれる、`#!/usr/local/bin/perl` のような行）があるの
ですが、これが今のサーバーには存在しないPerlの場所を指した
**古い地図**になっていました。

さらに、もう一つ厄介な問題がありました。この1行目の改行コードが、
Windows由来の「CRLF」という形式のまま残っていたのです。
人の目には見えない、ほんの小さな制御文字（`\r`）ですが、
サーバーからすると「その場所にPerlは無い」と誤解する原因になり
ます。地図の行き先が古いうえに、地図自体に**目に見えない汚れ**が
ついていたようなものです。

設定リロードを挟んで何度か試しても直らなかった（14:22、14:25、
14:29と3回連続で失敗）のは、この2つの問題が両方揃って初めて
解決する種類のものだったからです。

（余談ですが、この段階で「本当にファイルの場所を正しく認識できて
いるのか」を確かめるための実験コードを足しています。詳しくは
末尾の「おまけ」で――実はこれ、今も現役です。）

### 第4話：ようやく動いた（14:35）

シバン行の行き先を今のPerlの場所に直し、改行コードもUnix式の
LFに揃えたところ――

```
***.***.***.*** - - [18/Jul/2026:05:35:21 +0000] "GET /cgi-bin/bbs.cgi HTTP/1.1" 200
```

（JSTで14:35）初めて`200`（成功）が記録されました。ここまでで、
「サーバーが探す場所」と「スクリプトが指す場所」の**両方の地図**
が、ようやく一致したことになります。

### 第5話：一度直ったのに、再起動したらまた不機嫌になった（14:54）

ここで一安心……と思いきや、サーバーを一度きちんと再起動した
直後、また一瞬だけ500エラーが再発しました。

```
AH01241: exec of '/var/www/html/cgi-bin/bbs.cgi' failed
```

考えられるのは、カウンタ機能などが書き込むためのフォルダ
（`cgi-bin`や`count`）の書き込み権限が、何かの拍子にまた足りない
状態に戻っていた、というものです。権限を777（誰でも書き込める
設定）に整えたところ、翌分にはまた成功に戻りました。

```
***.***.***.*** - - [18/Jul/2026:05:55:17 +0000] "GET /cgi-bin/bbs.cgi HTTP/1.1" 200
```

（JSTで14:55）

### 最終話：安定動作の確認（14:59〜15:13）

投稿（POST）を試しても成功。その後6回以上ページを開き直しても
すべて200で応答し、エラーの再発はありませんでした。カウンタや
掲示板トップページも問題なく表示されることを確認し、無事に
「作業終了」となりました。

## まとめ：CGIが動かないとき、だいたいこの4つを疑う

今回の一件は、実は**CGIあるある**の詰め合わせでした。似たような
症状に困っている人は、まずこの4つを疑ってみるとよいかもしれません。

1. **ソースコードがそのまま表示される** →
   そもそもCGIとして認識されていない（実行設定・拡張子の紐付け等）
2. **404（見つからない）** →
   サーバーが探す場所と、実際にファイルがある場所がずれている
3. **500（実行できない）＋「シバン」という言葉** →
   スクリプト1行目が指すプログラムの場所が間違っている、
   または改行コードがWindows形式のまま残っている
4. **500（実行できない）＋書き込み系の機能だけ失敗する** →
   ディレクトリの書き込み権限が足りない

> **とくに4番目のパーミッション（権限）は初心者が一番つまずき
> やすいポイントです。** 今回も実際に、
> ① `cgi-bin`ディレクトリの書き込み権限を変更したら動いた、
> ② `count`ディレクトリが無かったので新規作成し、権限を777に
> 設定したら動いた、という**2段階のパーミッション修正**でようやく
> カウンタ機能が動くようになりました。「コードは合っているのに
> なぜか動かない」と思ったら、まず権限を疑ってみてください。

そして今回、最初から最後まで貫いていた方針はただ一つです。

> **掲示板としてのロジックは直さない。直すのは、いつもサーバー側の設定。**

20年前のコードを令和のサーバーで動かす――それは「古いものを
そのまま未来に届ける」ための、地味だけれど大事な作業なのだと
思います。

## おまけ1：実はまだ現役の「杖」コード

第3話のところで触れた、原因切り分けのために足したこの3行。

```perl
use Cwd 'abs_path';
use File::Basename;
chdir dirname(abs_path($0));
```

「実行時のディレクトリがズレているのでは？」という仮説を確かめる
ための保険として足したものですが、正直に訂正します。**この保険、
今も現役です。** 試しにこの3行を削除すると、今の環境ではCGIが
動かなくなります。

つまりこれは「念のため足してみたけど不要だった飾り」ではなく、
「これが無いとまだ立てない、現役の杖」でした。裏を返せば――
実行時にCGIが本来のディレクトリを正しく認識できるように、
Apache設定側だけで完全に解決する、という部分は**まだ完了して
いない**、ということでもあります。

「サーバー側の整備だけでコードは一切触らない」という動態保存の
理想からすると、この1点だけはまだ理想形に届いていません。今後の
宿題として、正直に記録に残しておきます。
