# 20年前のCGI掲示板を、令和のサーバーで動かす話　第2章

> 前回：CGIは無事動くようになった。めでたしめでたし……のはずが、
> 「これが無いと動かない」謎の3行が、コードの中にひっそり
> 残っていた。

**[実動する掲示板](https://gikonekos.com/cgi-bin/bbs.cgi)**

## あらすじ（前回までの話）

`bbs.cgi`はApacheの設定・シバン・パーミッションを整えたことで、
無事動くようになった。ただし1つだけ、動態保存の理想（コードは
一切触らない）に反する部分が残っていた。

```perl
use Cwd 'abs_path';
use File::Basename;
chdir dirname(abs_path($0));
```

削除すると動かなくなる、正体不明の「杖」。今回はこの杖が
本当に必要なのかどうかを、最後まで突き止める話。

---

## モジュールを変えてみる

エラーログを見ると、動いている実行方式は`mod_cgid`
（`[cgid:error]`というタグで分かる）だった。`mod_cgid`は、
スレッド型のサーバーでもCGIを軽く動かすための仕組みで、常駐する
「デーモン」がCGIを実行する。一方、昔ながらの`mod_cgi`は、
リクエストのたびにサーバー本体が直接CGIを立ち上げる、より
素朴な仕組みで、**スクリプト自身のディレクトリに移動してから
実行する**という気の利いた動作をする。

このサーバーはスレッド不要の「prefork」という方式で動いていた
ので、わざわざ`mod_cgid`である必要はない。そこで、こう切り替えて
みることにした。

```bash
sudo a2dismod cgid
sudo a2enmod cgi
sudo apachectl configtest
sudo systemctl restart apache2
```

設定チェックは`Syntax OK`。幸先が良さそうに見えた。

## 最初のつまずみ（実は自分で仕掛けた罠だった）

さっそく確認。

```bash
curl -s -o /dev/null -w "%{http_code}\n" http://127.0.1.1/cgi-bin/bbs.cgi
```

結果は**500**。エラーログを見ると、こう出ていた（17:07 JST）。

```
AH01215: (2)No such file or directory: exec of '/usr/lib/cgi-bin/bbs.cgi' failed: /usr/lib/cgi-bin/bbs.cgi
```

一瞬「`mod_cgi`への切り替えが失敗したのか」と思ったが、よく見ると
パスが`/usr/lib/cgi-bin/`——**Ubuntu標準の場所**に戻ってしまって
いる。実際に`bbs.cgi`があるのは`/var/www/html/cgi-bin/`のはず
なのに。

原因を辿ると、実はこれ以前の作業で、カスタムの`ScriptAlias`が
**httpsの設定ファイル（`000-default-le-ssl.conf`）にしか
書かれていなかった**ことが判明した。httpの設定ファイル
（`000-default.conf`、80番ポート）には元々そのカスタム設定が
無かったのだ。つまり、`http://`でアクセスするとUbuntu標準の
場所にフォールバックしてしまう——これは`mod_cgi`への切り替えとは
まったく関係ない、以前からの設定の抜け漏れだった。

試しに`https://`でアクセスしてみると――

```bash
curl -sk -o /dev/null -w "%{http_code}\n" https://127.0.1.1/cgi-bin/bbs.cgi
```

**200**。`mod_cgi`への切り替え自体は、ちゃんと成功していた。
「新しい変更のせいだ」と決めつける前に、まず何が本当に変わったのか
を確認する大切さを学んだ瞬間だった。

## コメントアウトしたつもりが、構文ごと壊していた

いよいよ本題。例の3行をコメントアウトしてみる。ここで、

```perl
/*use Cwd 'abs_path';
use File::Basename;
chdir dirname(abs_path($0));*/
```

という書き方をしてしまった。C言語やPHPではおなじみの
「`/* ... */`で囲むブロックコメント」だが、**Perlにはこの記法が
存在しない**。Perlにとって`/`は割り算や正規表現の記号なので、
これはコメントではなく意味不明な文として扱われ、思わぬところ
（20行目あたり）で構文エラーが連鎖した（17:15 JST）。

```
Scalar found where operator expected (Missing semicolon on previous line?) at ... line 20, near "$cgiurl"
syntax error at ... line 20, near "$cgiurl "
Execution of ... aborted due to compilation errors.
```

Perlで複数行をコメントアウトする時は、素直に各行の頭に`#`を
付けるのが正解。

```perl
#use Cwd 'abs_path';
#use File::Basename;
#chdir dirname(abs_path($0));
```

書き直して保存・反映すると、`curl`は無事**200**。5回連続で試しても
すべて200で、安定して動作した。「言語ごとにコメントの書き方が
違う」——初心者はもちろん、複数の言語を行き来していると
うっかりやりがちなミスだった。

## 本当の壁その1：「メッセージ読み込みに失敗しました」

curlでは200でも、実際にブラウザで投稿しようとすると、こんな
表示が出た。

> メッセージ読み込みに失敗しました

コードを見ると、原因はここだった。

```perl
open ( FLOG, "+<$logfilename" ) || &prterror ( 'メッセージ読み込みに失敗しました' );
```

`"+<"`は「既存のファイルを読み書き両用で開く」というPerlの書き方。
このファイル（`bbs.log`）のパーミッションが**664**（所有者と
グループしか書き込めない）になっていたため、Apacheの実行ユーザー
（`www-data`）が書き込めず、openそのものが失敗していた。

```bash
chmod 666 bbs.log
```

これで解決。

## 本当の壁その2：「過去ログ出力に失敗しました」

投稿はできるようになったが、今度は過去ログ機能でエラー。

> 過去ログ出力に失敗しました

これは2つの原因が重なっていた、少しやっかいなケースだった。

**原因A：ディレクトリの権限不足**

過去ログを保存する`log`ディレクトリが、`drwxr-xr-x`
（755、所有者は`ubuntu`）のままだった。Apacheの実行ユーザー
`www-data`は、このままでは新しいファイルをこのディレクトリの
中に作れない。777に変更して解決。

**原因B：スラッシュ1文字の有無で、パスがまるごと崩れる**

こちらはコード側の作りに関わる、少し本格的な話。過去ログの保存先
パスを組み立てている部分がこうなっていた。

```perl
# 過去ログの保存方法
#   0 : 日毎
#   1 : 月毎
$oldlogsavesw = 1;   # ← 月毎モード

...

if ( !$oldlogsavesw ) {
    # 日毎モード：書式に "/" が入っている
    $oldlogfilename = sprintf ( "%s/%d%02d%02d.$oldlogext",
      $oldlogfiledir, $year, $mon, $mday );
} else {
    # 月毎モード：書式に "/" が入っていない！
    $oldlogfilename = sprintf ( "%s%d%02d.$oldlogext",
      $oldlogfiledir, $year, $mon );
}
```

今回のサイトは「月毎モード」。この分岐の書式には`/`が
**入っていない**。つまり、ディレクトリ名（`$oldlogfiledir`）
自身が末尾にスラッシュを持っていないと、ディレクトリ名とファイル名が
そのままくっついてしまう。

- `$oldlogfiledir = './log'`（スラッシュ無し）→
  `./log` + `2026` + `07` + `.dat` = **`./log202607.dat`**
  （`log`ディレクトリの**中**ではなく、cgi-binの直下に
  ファイルができてしまう）
- `$oldlogfiledir = './log/'`（スラッシュ有り）→
  `./log/202607.dat`（正しい場所）

実は以前の作業で「末尾のスラッシュを試しに消してみる」という
判断をしたことがあり、それが今回のこの症状を引き起こしていた。
スラッシュを戻し、上記の権限（原因A）もあわせて直したところ、
過去ログがようやく正しい場所に、正しい形で出力されるようになった。

## 決着：杖は、もう要らなかった

ここまでの3つの修正（コメントの書き方・`bbs.log`の権限・
`log`ディレクトリの権限とスラッシュ）を終えた状態で、改めて
確認した。

- 投稿：成功
- 過去ログ一覧：`2026/07`のログが表示される（226 bytes）
- curlの連続アクセス：安定して200

**そしてこの間、例の3行はずっとコメントアウトしたまま
（＝実質削除）だった。**

つまり――

> `mod_cgid`から`mod_cgi`に切り替えたことで、当初の仮説通り
> chdirは本当に不要になっていた。ただし、それが見えるまでに、
> 全く別の3つの落とし穴（コメントの構文ミス・bbs.logの権限・
> logディレクトリの権限とスラッシュ）が立ちはだかっていた。

「chdirが要る/要らない」というシンプルな二択に見えた問題は、
実際には**複数の独立した不具合が同時に存在していた**ために、
なかなか白黒つかなかった、というのがこの章の結末である。原因を
1つ直すたびに別の症状が顔を出す——地味だが、こういうものだと思う。

## おまけ：配布パッケージには「空のbbs.log」が要る、という仕様

後日、Windows 11の環境でも同じ検証を行った際に判明した、地味だが
重要な仕様がある。

> **zipをダウンロードして解凍し、ファイルをアップロードしただけの
> 初期状態では、ルートに空（0バイト）の`bbs.log`が置かれていないと、
> エラーで停止する。**

これはバグではなく、コードの作り上そうなる**仕様**である。理由は
この章の「壁その1」で見た、まさにこの1行にある。

```perl
open ( FLOG, "+<$logfilename" ) || &prterror ( 'メッセージ読み込みに失敗しました' );
```

Perlの`"+<"`（読み書き両用オープン）は、**既に存在するファイルしか
開けない**。`">"`や`"+>"`と違い、ファイルが無い場合に新規作成は
してくれない。つまり、配布用のzipに`bbs.log`という空ファイルを
あらかじめ同梱しておかないと、初回アクセスの時点で必ずこの
エラーで止まってしまう。

古いCGI掲示板を配布・保存する際は、コードだけでなく「空の
`bbs.log`」もセットで残しておく必要がある、ということを、
記録として残しておく。
